Why now?

Three things changed in the last 18 months:

  • Replication compliance is no longer optional. AEA Data Editor since 2019; JF Disclosure Statement since 2023; JAR full-data archive expected by 2027. Desk rejects on package failures are rising.
  • Referee reports got harder, not easier. Higher methods bar at AER / JF / JAR; pre-registration creep into observational work; “what would change my mind” becoming a referee-side norm.
  • Agentic AI matured fast. “Tool-using” went from research preview (2024) to production (2025). The gap between chatbot and agent is now the same as the gap between spreadsheet and programming language.

  • High-stakes papers, slow review cycles, replication-package compliance now part of the job, not a footnote.

What “agentic” actually means

A chatbot:

You: “Edit the abstract to mention the panel-data dimension.” Bot: “Here’s a revision: [text]” You: copy → paste → save → repeat.

An agent:

You: “Edit the abstract to mention the panel-data dimension.” Agent: opens the file, reads the abstract, writes the edit, saves, runs the bibliography check, opens the result, reports back.

The difference isn’t intelligence. It’s the ability to act — and to verify it acted correctly.

The four agentic primitives

Primitive What it gives you
Tool use Read files, run commands, edit on disk, call APIs
Planning Break a task into reviewable steps before acting
Sub-agents Spawn focused workers (proofreader, code reviewer, fact-checker) in parallel
Hooks Deterministic enforcement at lifecycle events (pre-commit, post-edit)

The next four parts walk through these one at a time. Today’s lecture in one sentence: a workflow is the pattern that connects them.

CLAUDE.md as control center

The single most important file in this stack

CLAUDE.md lives at your project root.

It loads automatically at the start of every Claude session.

It tells the agent what your project is, what the rules are, and what good looks like — before the agent does anything.

It’s ~150 lines. It changes everything downstream.

What’s in mine

# CLAUDE.MD — Causal Panel Data Course

## Core Principles
- Plan first; verify after.
- Single source of truth: Beamer .tex is authoritative.
- Quality gates: nothing ships below 80/100.

## Folder Structure
- Slides/ — Beamer .tex
- Quarto/ — RevealJS .qmd
- scripts/R/ — analysis pipeline
- Bibliography_base.bib — centralized citations

## Commands
TEXINPUTS=../Preambles:$TEXINPUTS xelatex -interaction=nonstopmode file.tex
quarto render file.qmd

I’ll open the actual file in a moment.

What that gives Claude

  • Folder structure → knows where to put things, what to read
  • Commands → knows the exact LaTeX flags, knows quarto render over pandoc
  • Quality thresholds → won’t ship below 80/100 score; will halt and ask
  • Source-of-truth contract → won’t edit Quarto without syncing Beamer

Live walkthrough — the actual CLAUDE.md

[In live demo: Pedro opens claude-code-my-workflow/CLAUDE.md in editor. Walks line by line. Audience sees that the file is short, specific, and prescriptive about what Claude can’t guess.]

Demo prompt

After the tour: delete the ## Commands section. Re-open Claude. Ask: “render the slides.” Watch Claude guess pandoc (wrong tool). Restore the section. Ask again. Watch Claude pick quarto render. The file is the difference.

Three sections that always pull weight

  1. Commands you’d type — exact CLI invocations Claude can’t infer (build flags, tool versions, environment variables).
  2. Folder structure — where things live, with one-line “what’s here” notes per directory.
  3. Quality thresholds + source-of-truth contracts — non-negotiables. “Never edit X without syncing Y.”

Three sections that look helpful but aren’t

  • ❌ A list of every package you use. Claude knows R packages.
  • ❌ “Write clean code.” Claude knows.
  • ❌ A re-summary of files Claude can read. Skip — Claude reads them.

Test: if removing the line caused Claude to make a mistake, keep it. If not, cut it.

The CLAUDE.md test, formalized

# 1. Open CLAUDE.md
# 2. Comment out a line
# 3. Restart Claude session
# 4. Ask: "render the slides" / "run the analysis" / "fix this bug"
# 5. Did Claude get it wrong?
#    YES → keep the line
#    NO  → delete it

This isn’t a metaphor — it’s a debugging method. Run it on your own CLAUDE.md once a quarter.

Effective CLAUDE.md

✅ Include / ❌ Exclude

✅ Include ❌ Exclude
Bash commands Claude can’t guess Standard language conventions
Code-style rules that differ from defaults Detailed API docs (link to source)
Testing instructions + preferred runners Self-evident practices (“write good code”)
Repository etiquette (branch naming, PR conventions) File-by-file descriptions
Architectural decisions specific to your project Information that changes frequently
Developer-environment quirks (env vars, paths) A summary of files Claude can already read
Common gotchas / non-obvious behaviors Long explanations or tutorials

Specificity wins

  • ❌ “Use good R code style.”
  • ✅ “Use fixest::feols, not lm. Cluster SE at firm level. set.seed(20260427) once at top of 00_run_all.R.”
  • ❌ “Be careful with citations.”
  • ✅ “Citations: \citep{key} for parenthetical, \citet{key} for narrative. All keys must exist in Bibliography_base.bib before commit.”

Specificity wins, part 2

  • ❌ “Write clean Python.”
  • ✅ “Use pandas for data, pyarrow for serialization. Type hints required for public functions. Black formatter. pytest -xvs is the test runner.”
  • ❌ “The folder structure is intuitive.”
  • ✅ “data/raw/ — never modified after download. data/clean/ — outputs of 02_clean.R only. results/ — read-only, regenerated by 00_run_all.R.”

Length budget

  • Aim for 100-200 lines. That’s where most working CLAUDE.md files live.
  • Hard ceiling: ~300. Past that, important rules get lost.
  • If yours is over 300: something belongs in a rule file (path-scoped), not the universal CLAUDE.md.

Rules and hooks (preview)

When CLAUDE.md gets too long

The fix: path-scoped rule files under .claude/rules/.

  • A rule file is just like CLAUDE.md, but loads only when Claude touches a matching file.
  • .claude/rules/r-code-conventions.md with paths: "*.R" → loads only when an .R file is in play.
  • Keeps CLAUDE.md focused on universal stuff. Specialty knowledge waits for its moment.

A rule example

# .claude/rules/replication-protocol.md
---
paths: ["scripts/**/*.R", "scripts/**/*.do", "scripts/**/*.py"]
---

# Replication-First R Code

When asked to extend an analysis:
1. FIRST: replicate the original numbers in the manuscript.
2. THEN: extend.
3. NEVER: ship a script that produces numbers that don't match the paper.

If the original numbers don't replicate, STOP and report what differs.

This rule auto-loads when Claude opens any .R / .do / .py file. Outside that scope, the rule isn’t in context — staying lean.

Hooks: deterministic enforcement

CLAUDE.md and rules are advisory — Claude reads them and (usually) follows them.

Hooks are scripts that fire deterministically at lifecycle events. They don’t ask Claude — they enforce.

  • PreToolUse — before any tool runs (block dangerous patterns)
  • PostToolUse — after tool completes (sanity check, log, notify)
  • Stop — when Claude is about to stop responding (verify, remind)
  • PreCompact — before context auto-compaction (save plan to disk)

A hook example

# .claude/hooks/log-reminder.py — Stop hook
# Fires when Claude is about to stop responding.
# If session log hasn't been updated in N responses, remind to update it.

if responses_since_last_log_update > 50:
    print("Session log is stale. Update quality_reports/session_logs/", file=sys.stderr)

This isn’t AI judgment — it’s a Python script. Deterministic. Runs every Stop, no exceptions.

The full lifecycle (preview)

Step What happens Time mark
1 I paste the prompt. Claude enters plan mode. T+0
2 Plan appears. I narrate. T+3
3 I approve. Implementation begins. T+5
4 Agent reads, edits, self-verifies. T+6
5 Multi-agent review fans out (proofreader + domain-reviewer + claim-verifier). T+12
6 Review reports surface. T+18
7 Fixes applied. T+21
8 Quality score, summary, done. T+25

What I want you to notice

  1. Plan mode is read-only. Claude can’t edit until I approve. This is the safety boundary.
  2. The review fanout is parallel. Three agents in three contexts — no cross-contamination.
  3. The verifier never sees my prompt. It checks the work in a fresh context (Chain-of-Verification, Dhuliawala et al. 2023).
  4. The score is from a deterministic Python script, not the agent’s self-evaluation. Numerical, reproducible.

Part 3 — Building Reusable Skills

What is a skill, really?

  • A skill is a Markdown file at .claude/skills/<name>/SKILL.md.
  • It has YAML frontmatter (when does Claude trigger it?) and a body (what to do).
  • It auto-loads when a relevant prompt is detected — or you invoke it explicitly with /<name>.
  • That’s it. No code, no installation, no plugin system.

A skill is just a written instruction — but a triggerable, version-controlled, reusable one.

What a skill is NOT

  • ❌ A plugin (no compilation, no extension API)
  • ❌ A chat template (skills can read files, run code, call other skills)
  • ❌ A prompt library entry (those are static; skills run workflows)
  • ❌ Magic (you’ll see exactly what’s in one in 90 seconds)

Anatomy of a skill

---
name: my-skill-name
description: One-sentence description with trigger keywords. "Use when user says X, Y, Z."
argument-hint: "[optional flags / inputs]"
allowed-tools: ["Read", "Write", "Bash", "Grep", "Glob"]
disable-model-invocation: false   # set true if this skill must be user-explicit only
---

# My Skill Name

## Workflow

### Phase 1 — Inputs
Read the spec, validate inputs, refuse if results are present.

### Phase 2 — Generate
Produce the output, MUST/SHOULD/MAY annotated.

### Phase 3 — Verify
Cross-check against domain rules.

### Phase 4 — Output
Write to disk. Report to user.

That’s a real skill template. Frontmatter ≤ 6 fields. Body = numbered phases.

Why this beats writing the workflow into your prompts

  • Reusable — write once, invoke anytime, share with co-authors
  • Versionable — git diff your skill the way you git diff your code
  • Auditable — anyone reading your repo sees the workflow as text
  • Composable — a skill can invoke another skill (e.g., /preregister calls /verify-claims)

Walk-through: /preregister

What it does (one slide)

/preregister drafts a registry-ready preregistration document in your choice of style:

  • OSF (default) — full template, social-science default
  • AsPredicted — 9-question short form
  • AEA RCT Registry — required for AEA-journal field experiments since 2018

It refuses to draft if your description contains results — preregistration is forward-looking by definition.

Demo prompt (verbatim)

I’m planning a 2x2 between-subjects survey experiment on N=800 US adults via Prolific. Manipulating message-source partisanship × policy framing on attitudes toward climate policy. Primary DV is a 4-item attitude index. Hypothesis: in-party + gain framing produces highest support. Pre-registered analysis: two-way ANOVA. Run /preregister –style aspredicted on this.

Why this skill, for this audience

  • Pre-registration is the #1 referee ask in finance experiments (JF, RFS) and accounting field/lab work (TAR, JAR, JAE).
  • AEA RCT Registry is mandatory for AEA-journal field interventions; many B-school field studies don’t realize they qualify.
  • The MUST/SHOULD/MAY annotation discipline is exactly the framing reviewers expect — directional hypothesis, named estimator, ex-ante exclusions, sample-size stopping rule.

Three things to notice in the SKILL.md

  1. Trigger phrases in the description — “preregister”, “OSF”, “AsPredicted”, “AEA RCT”, “PAP”, “preanalysis plan”. Why so many? See Effective triggers below.
  2. disable-model-invocation: true — this skill writes a commitment document; it must be user-intent.
  3. Cross-check phase — refuses to mark “ready” if hypothesis isn’t directional, estimator isn’t named, exclusions aren’t ex-ante.

Effective trigger descriptions

A skill triggers when its description matches the user’s prompt. So the description must contain the words your future self / your co-author would actually use.

  • ❌ description: A preregistration tool. — too generic; never triggers
  • ❌ description: Pre-registers a study. — singular, formal; misses real usage
  • ✅ description: Draft a preregistration document (OSF, AsPredicted, AEA RCT Registry style). Use when user says "preregister", "draft a preregistration", "OSF preregistration", "AsPredicted", "AEA RCT registry", "PAP", "preanalysis plan".

The description IS the trigger. Pack it with the actual phrases people use.

Skills vs MCPs (a quick aside)

A common question: “How is a skill different from an MCP?”

  • A skill is a Markdown instruction Claude reads in-context. Triggered by user phrases. Lives in your repo as a .md file.
  • An MCP (Model Context Protocol) is a runtime tool integration — a separate process that connects Claude to external services (databases, APIs, file systems, Notion, JIRA, your home Postgres).

Different layer of the stack. Skills tell Claude how to think; MCPs give Claude more things to talk to.

When to reach for which

Scenario Use a skill Use an MCP
“Format my regression tables in JF style” ✅ —
“Pull our latest WRDS query into the analysis” — ✅
“Draft a preregistration document” ✅ —
“Update my Notion ‘papers under review’ page” — ✅
“Audit reproducibility against the R script” ✅ —
“Query our shared Postgres database of working-paper revisions” — ✅

Most academic workflows are skill-heavy. MCPs become valuable when you connect Claude to live external data you don’t already have in your repo. Further reading: ai-mba.io/tutorials/skills-vs-mcps.

Design principles

Reusability

  • Specific enough to do something useful, general enough to apply more than once.
  • Trap: writing a skill that only solves this paper’s problem. Re-frame to the class of problems.
  • Test: would your co-author / next-year-you actually invoke this?

Specificity (in the description AND the body)

  • In the description: trigger phrases people actually say, examples of when to use it, when not to use it.
  • In the body: literal commands, exact filenames, named estimators. “Run an analysis” → no. “Run Rscript scripts/02_main.R and read the output of Sys.time()” → yes.

Built-in verification

The best skills don’t just do — they check.

  • /preregister refuses to mark MUST sections complete if hypothesis isn’t directional.
  • /audit-reproducibility (Part 5) compares numerical claims to script outputs with tolerance thresholds.
  • /checkpoint reads git status to verify state before writing the snapshot.

A skill without a verification step is a fancy macro. A skill with one is a workflow.

Cost-of-creation, honestly

  • A first skill: 30-60 minutes to draft, test, iterate.
  • A polished skill: 2-4 hours including verification, examples, edge-case handling.
  • A production skill (used by colleagues): half a day for full doc + tests.

If you’re spending less than 30 min, you’re under-investing in the description. If more than 4 hours on draft #1, you’re over-engineering.

Cumulative leverage

  • Year 1: 5 skills. Each saves 1 hour/month. Net: 60 hours/year freed.
  • Year 2: 12 skills. Each saves 1-2 hours/month. Net: 200+ hours.
  • Year 3: 20+. Skills start invoking each other. Net: hard to measure — your default workflow becomes “Claude + my repo of skills”.

The leverage compounds because each skill is a written commitment to how you do this kind of work. Future-you can read it.

Part 4 — Orchestration + Multi-Agent

The honest framing

  • Most “AI for research” demos online ship the first agent output.
  • That’s not a workflow — it’s a draft.
  • The leverage comes from running the loop: implement → verify → review → fix → re-verify → score.

A model alone is impressive. A model in a loop is useful.

The 6-step orchestrator loop

1. IMPLEMENT   — agent does the work
2. VERIFY      — does it compile / render / parse?
3. REVIEW      — what does a focused critic think?
4. FIX         — apply review findings
5. RE-VERIFY   — did fixes break anything?
6. SCORE       — numerical quality threshold
       │
       ├── Score ≥ threshold → ship
       └── Score < threshold → loop back to step 3 (max 5 rounds)

The point isn’t the specific 6 steps. It’s that steps 2 and 5 have to be deterministic.

Where this lives in the template

  • The orchestrator is a pattern, not a daemon. There’s no background process running it.
  • Specific skills implement it: /commit, /qa-quarto, /review-paper --adversarial, /slide-excellence, /create-lecture, /data-analysis, /review-paper --peer.
  • Plan approval does NOT auto-trigger the loop. You invoke a skill; the skill runs the pattern within its own scope.

Honesty matters: this is a contract that skills implement, not a magic background service.

When agents help

Three patterns where multi-agent earns its keep

  1. Parallel independent work — three papers to summarize, ten regression specs to estimate, six sections to proofread. No coordination needed; each agent reads its file and reports back.
  2. Critic + fixer pairs — one agent finds problems (read-only); a different agent applies fixes. They alternate until the critic says “approved.” Eliminates the “I just wrote this code, it looks fine to me” trap.
  3. Fresh-context verification — a verifier agent that has never seen your draft answers verification questions from the source material alone. It can’t self-confirm a hallucinated claim because it doesn’t know the claim was made.

Pattern 1: parallel independent work

  • Three referee reports come in. You spawn three agents in parallel: one per report.
  • Each reads its assigned report + the manuscript. Returns a structured response sketch.
  • You synthesize.

When it works: the tasks are genuinely independent. When it breaks: the tasks need to share context (then it’s serial, not parallel).

Pattern 2: critic-fixer loop

  • The critic reads the artifact (Beamer source, Quarto target, manuscript section). Read-only — it cannot edit.
  • The fixer receives the critic’s findings and applies them.
  • They alternate. Critic re-audits after each fixer pass. Loop until APPROVED (or max 5 rounds).

Why two agents and not one: the agent that wrote the fix shouldn’t be the one judging it. Role separation = honest review.

Pattern 3: fresh-context verification (CoVe)

  • The verifier never sees the draft. It only sees: claims, verification questions, source material.
  • Architecturally enforced via context forking — the verifier cannot know what was originally written.
  • Adapted from Dhuliawala et al. 2023, Chain-of-Verification Reduces Hallucination in LLMs (arXiv:2309.11495).

Why this matters for B-school work: referee reports cite specific papers. CoVe catches misattributions BEFORE you submit.

Three patterns, three failure modes

Pattern Best for Failure mode
Parallel Genuinely independent tasks Tasks need to share context — coordination chaos
Critic-fixer Surface-level + structural review Critic and fixer are the same agent — “self-confirmation”
Fresh-context CoVe Factual claims (citations, numerical facts) Forgetting to fork the verifier’s context — verifier confirms what it just generated

When multi-agent doesn’t help

The decomposition trap

  • “Spawn 5 agents to refactor this codebase” — sounds good, fails badly.
  • Each agent works on a slice; slices interact; no agent has the whole picture.
  • You get 5 partial fixes that conflict and an integration nightmare.

Rule: multi-agent is for tasks where the decomposition is clear. If you can’t write the decomposition in 2 sentences, don’t fan out.

A concrete failure mode

  • I once asked: “spawn 3 agents to add fixed effects to my regression in three different ways.”
  • Agent 1: firm FE. Agent 2: industry × year FE. Agent 3: matched-sample.
  • Each rewrote the script in its own image. Three branches. Two days reconciling.
  • I should have run them serially — one branch, one decision per round.

The lesson: parallel ≠ better. Parallel = right for independent work; serial = right for interacting work.

When ONE agent is enough

  • Most editing tasks. One agent + the orchestrator loop is fine.
  • Most analysis tasks. Multi-agent on a regression script is over-engineering.
  • Quick fixes. The cost of fanning out exceeds the benefit if the task takes < 5 minutes.

Heuristic: add a second agent only when you can articulate exactly what that agent sees that the first one couldn’t.

Part 5 — Research + Replication

A short personal anecdote

I had a working paper rejected once because a single number in Table 3 didn’t match the .csv I shipped. I spent 8 hours figuring out why.

The skill I’m about to show you would have caught it in 90 seconds.

Why this part lands hardest for B-school faculty

  • AEA Data Editor has been operational since 2019. Every accepted AEA paper goes through a numerical reproduction check.
  • JF Disclosure Statement (2023+) requires data + code at acceptance. Verification is editorial.
  • JAR replication archive is rolling out. Soon every accepted paper.
  • TAR, RFS, JFE are following.

This is no longer “good practice.” It’s increasingly a desk-reject hazard.

What the AEA Data Editor checks

  • Every numerical claim in the manuscript must reproduce from the shipped code + data.
  • Tolerance: typically 4 decimal places on point estimates; exact match on N, R², degrees of freedom.
  • Discrepancies → major revision required to fix — even after acceptance.

You don’t want to discover the discrepancy after the editor’s email.

Replication-first as a workflow rule

The repo’s .claude/rules/replication-protocol.md says:

When asked to extend an analysis: FIRST replicate the original numbers. THEN extend. NEVER ship a script that produces numbers that don’t match the paper.

This rule auto-loads on *.R, *.do, *.py. Claude reads it at the start of every analysis session.

It’s a rule, not a hook — Claude can override with reason. But the default is replication-first.

Code inventory + comparison

The dependency graph

manuscript.tex ──cites──> Table 2
Table 2        ──from──> scripts/R/_outputs/results.rds
results.rds    ──by──> scripts/R/03_analyze.R
03_analyze.R   ──reads──> data/clean.rds
clean.rds      ──by──> scripts/R/02_clean.R
02_clean.R     ──reads──> data/raw/

A bug in 02_clean.R invalidates Table 2. Reviewing the manuscript without touching the code misses this class of error.

What /review-paper does about it

  • Detects script references in the manuscript (\input{scripts/...}, %% source: scripts/..., table names matching script outputs).
  • Auto-invokes /review-r on each referenced script (code-quality review).
  • Auto-invokes /audit-reproducibility on the paper ↔︎ outputs pair (numeric verification).
  • Surfaces cross-artifact findings at the top of the review report.

This isn’t a feature you trigger. It’s the default for /review-paper when scripts are detected.

AEA-style packaging

What an AEA-compliant package looks like

replication-package/
├── README.md                  # data sources, software versions, runtime estimate
├── 00_run_all.R               # ONE entry point that produces all outputs
├── data/
│   ├── raw/                   # downloaded, never modified
│   └── clean/                 # outputs of cleaning step
├── scripts/
│   ├── 01_load.R
│   ├── 02_clean.R
│   ├── 03_analyze.R
│   └── 04_figures.R
├── outputs/
│   ├── tables/                # match Table N in manuscript
│   └── figures/               # match Figure N in manuscript
└── sessionInfo.txt            # captured at the end of 00_run_all.R

What the workflow gives you

  • /data-analysis scaffolds this structure on a new project.
  • Numbered scripts (01_*, 02_*, etc.) — implicit dependency order; AEA Data Editor loves it.
  • 00_run_all.R is the entry point — a Data Editor runs it once, no flags, expects matching outputs.
  • sessionInfo.txt captured automatically = R version + package versions reproducible.

You don’t have to remember any of this. The skill enforces the structure.

What the rule enforces

.claude/rules/r-code-conventions.md (v1.6.x) Section 8 — Numerical Discipline:

  • No floating-point == comparison
  • CDF clamping with named epsilon (eps <- 1e-12)
  • Integer literals for counts (1L, not 1)
  • Pre-allocated vectors
  • Deterministic bootstrap seeding (set.seed(YYYYMMDD) once at the top)
  • Explicit na.rm (no defaults)
  • No T/F shorthands

These are the kinds of bugs that produce “the script ran but the numbers are slightly off” — which is exactly what AEA Data Editors catch.

A bug class /audit-reproducibility catches that I’d miss

  • I write subset(df, year > 2010) in the cleaning script.
  • Manuscript reports N = 14,562.
  • Three months later, I add & industry != "financial" to the subset for a robustness check.
  • Forget to update the manuscript; new N is 14,489 — manuscript still says 14,562.
  • I’d never catch this on re-read. The skill catches it in 5 seconds.

Part 6 — Refereeing + Review Simulation

What if every paper got a peer review before you submitted it?

  • Not in the “I emailed it to my co-author” sense.
  • An editor who selects two referees with different perspectives, who write structured reports, and an editorial decision with an explicit revision plan.
  • Twenty minutes. Not weeks.

That’s what we’re about to show.

Why this works (and where it doesn’t)

  • ✅ Surface-level + structural issues: missing citations, unclear identification, weak first-stage, framing drift.
  • ✅ Referee-perspective stress-tests: a “credibility skeptic” reads it differently than a “policy relevance” reviewer.
  • ❌ Substantive contribution. The simulation can’t tell you the result is interesting. That’s the editor’s actual job.
  • ❌ Field-specific norms beyond what’s in the journal profile.

Use it as a dress rehearsal, not a substitute for real referees.

The 6-way disposition taxonomy

Every B-school referee has a type. We model them explicitly.

Disposition Reads it asking…
STRUCTURAL “Is the model specification right?”
CREDIBILITY “Is identification airtight?”
MEASUREMENT “Are the variables the right things?”
POLICY “Does this matter outside of academia?”
THEORY “Where’s the mechanism?”
SKEPTIC “What’s the strongest case against this?”

Two referees with deliberately different dispositions stress-tests the paper from non-overlapping angles.

Journal calibration

The editor reads .claude/references/journal-profiles.md and adjusts:

  • Disposition pool weights per journal (AER skews CREDIBILITY + POLICY; APSR skews THEORY; AJPS skews CREDIBILITY + MEASUREMENT)
  • Domain-referee dimension weights (AER raises Contribution; QJE raises Identification)
  • Methods-referee paper-type rubrics (formal-theory weights model-novelty; survey-experiment weights design + balance)

A simulated AER review should feel different from a simulated AJPS review. The journal is the prior.

What we ship

In claude-code-my-workflow/.claude/references/journal-profiles.md (v1.8.0):

  • Econ top-5: AER, QJE, JPE, ECMA, ReStud
  • Political science top-3: APSR, AJPS, JOP (added v1.8.0)
  • Coming for ND talk: JF + JAR (v1.8.1, this week)
  • Sociology, public health, psych: template + skeleton — your fork-and-extend

Each profile is ~40 lines. Adding your home journal: 30 minutes once.

“What would change my mind?”

Every MAJOR concern in a referee report from this pipeline must include:

What would change my mind: [specific test, estimator, robustness check, or evidence that would resolve this concern]

  • This is a discipline, not a feature. Without it, “I’m worried about endogeneity” is taste; with it, “I’d be convinced if you ran an IV with X as instrument” is actionable.
  • Same discipline for both referee agents (domain + methods).
  • Without this constraint, the simulated reports drift toward un-actionable adversarialism.

Drafting referee responses

The flip side

You just saw a simulated peer review. The other half of the workflow:

Now I have a real referee report I disagree with. How do I respond?

/respond-to-referees parses a referee report and maps each concern to a specific revision in the manuscript.

What it produces

  • Concern-by-concern table: referee text → classification (Addressed / Partial / Deferred / Disagreement) → manuscript location of the response.
  • Drafted response prose for each concern, polite-but-firm, in the project’s voice.
  • A coverage summary at the top: X addressed, Y partial, Z deferred, W disagreement.
  • Revision-letter scaffold ready to upload.

Why classification matters for R&R

  • Addressed — you did the thing. Cite where.
  • Partial — you did some of the thing. Cite where; explain what’s missing and why.
  • Deferred — you didn’t do the thing this round. Explain why deferred (out of scope, future work, infeasible).
  • Disagreement — you don’t think the thing should be done. Explain courteously and substantively.

Editors at JF / JAR / TAR want to see all four categories used appropriately. Avoiding “Disagreement” looks evasive.

A concrete example

A referee writes:

“The first-stage F-statistic of 9 is well below the conventional weak-instrument threshold. The IV results should not be interpreted causally.”

/respond-to-referees produces:

Classification: Addressed. Where: Manuscript p. 14, footnote 22; Online Appendix B.3. Response: “We thank the referee for raising the weak-instrument concern. We now report Anderson-Rubin confidence intervals (Online Appendix B.3) which are robust to weak instruments; the result is qualitatively similar with much wider intervals. We have added a footnote on p. 14 noting the limitation explicitly.”

Alternative: strategic-revision (jusi-aalto)

A community Claude skill built specifically for our audience. Different shape than ours.

  • Where /respond-to-referees produces concern-by-concern drafted responses, jusi-aalto/strategic-revision treats reviewer requests as a directed acyclic graph.
  • It identifies dependencies (e.g., “Referee 2’s concern about the IV depends on resolving Referee 1’s concern about sample selection first”), validates the DAG with NetworkX, and sequences work into execution blocks with co-author assignments.
  • Stated audience: empirical accounting + finance papers — exactly this room.

Different tool for different sub-task: /respond-to-referees for the prose; strategic-revision for the project plan. Use both.

Cross-tool strategies

Why use more than one AI tool

  • No single tool is best at everything. Claude shines on repo-context tasks; Codex shines on focused-completion problems; Gemini on multimodal.
  • A tool reviewing its own work has confirmation bias. A different tool, different training data, different failure modes — independent review.
  • Cost arbitrage. Cheap Haiku-class models for triage; Opus-class for the hard part; Codex for the tasks where it’s strictly better.

The orchestration story extends: multi-agent across tools, not just within Claude.

The pattern: Claude reviewing Codex

1. Codex generates a 200-line refactor.
2. Claude reads the diff in a fresh context (not the prompt that generated it).
3. Claude flags issues from a different angle than Codex would.
4. You see both views before deciding.

The plugin codex:adversarial-review (in claude-code-my-workflow) is exactly this. Different tool reviewing different tool’s work.

When cross-tool actually helps

  • Refactors with subtle correctness implications — different tools see different bugs.
  • Code in languages you don’t read fluently — your one tool agreeing with itself isn’t reassurance.
  • High-stakes one-shot tasks — you’d rather pay 2x to get 2x review than re-do it.

When it doesn’t

  • Routine work. Cost-of-orchestration > value-of-second-opinion. Use one tool, save the round trip.
  • Genuinely ambiguous decisions (taste, style). Two tools voting doesn’t tell you which is right.
  • Speed-critical iteration loops. Cross-tool adds 30 sec / iteration. On a 50-iteration day = 25 min lost.

Heuristic: cross-tool for one in ten tasks — the high-stakes ones. Not as a default.

  • Goldsmith-Pinkham × Princeton — Yale finance professor; 5-episode video series on Claude Code for applied economists (March 2026 at Markus’ Academy). The most authoritative public material on this topic for our field.
  • Claes Bäckman — Claude Code in VS Code for Academic Economists — practical written guide; ~3,500 words covering setup, R/Stata/Python/LaTeX extensions, CLAUDE.md, skills, git, performance. Recommended companion to this talk — written-form, IDE-focused.
  • hugosantanna/clo-author — authoring-side fork of claude-code-my-workflow. Theorist agents, 30+ journal profiles, worker-critic pairs. Source of /checkpoint and /review-paper --peer (with permission).
  • jusi-aalto/strategic-revision — DAG-based peer-review response planning for empirical accounting + finance R&R. Highly recommend for accounting faculty.
  • jusi-aalto/crossref — bibliography validation against Crossref API.
  • chrisblattman/claudeblattman — guide for non-technical academics. Less code, more workflow.

What others have built (broader)

  • scunning1975/MixtapeTools — Scott Cunningham’s TikZ + Rhetoric of Decks. Source of our TikZ infrastructure.
  • karpathy/autoresearch — Karpathy’s experiment in constraint-based autonomous research. Aspirational; different shape.
  • ai-mba.io — AI MBA curriculum and tutorials. Targets B-school audiences specifically. Their skills vs MCPs tutorial is a good follow-up to Part 3.

Anthropic-shipped utilities

Built into Claude Code itself, no fork needed:

  • /team-onboarding (docs) — package your local setup into a replayable guide for collaborators. Recommend for lab groups.
  • /autofix-pr — runs Claude Code on a GitHub PR, applying CI-style fixes
  • /powerup — interactive lessons (Anthropic’s tutorial system)
  • Ultraplan (docs) — cloud-based plan editing, sync to local
  • /loop self-pacing — recurring or self-paced background tasks
  • /less-permission-prompts — diagnose permission-prompt fatigue

Cross-tool: Codex plugin (Part 7’s earlier demo)

Built by OpenAI’s VB Srivastav and team. Source: OpenAI Developer Community announcement.

  • “It is Codex, just invoked from inside Claude Code” — reuses local Codex CLI auth, config, MCP setup.
  • Three use cases: (1) standard Codex review, (2) adversarial / skeptical review, (3) second-pass delegation when you want a different agent’s eyes.
  • We saw use case 2 in the optional Part 7 demo (/codex:adversarial-review).

Resources + start

Three things to do this week

  1. Clone Claude-Mini (the repo you’re looking at): github.com/pedrohcgs/Claude-Mini. Slides + all demo bundles.
  2. Fork claude-code-my-workflow: github.com/pedrohcgs/claude-code-my-workflow. The full template (30 skills, 14 agents, 24 rules, 6 hooks, v1.8.0).
  3. Write a CLAUDE.md for your next paper. 30 minutes. Highest-leverage action you can take.

If you do nothing else: #3. Open one paper directory, write 100 lines, and the whole rest of this stack pays off.

Three things to do this month

  1. Pick one repetitive task you do 3+ times a year. Write it as a skill. (Sample use cases on the handout.)
  2. Run /audit-reproducibility on your most recent submitted paper. Find the issue. Fix it.
  3. Try /review-paper --peer <your-target-journal> on a current draft. Treat the simulated review as a dress rehearsal.
